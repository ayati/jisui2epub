# 横書き本対応 設計書（Phase 1: 最低限対応）

作成: 2026-07-20 / 改訂: 2026-07-20（実本サンプルによる検証を反映）
対象: jisui2epub.py（現行 main ブランチ bc7ce4e 時点）
検証サンプル: `temp_sample/30秒でわかる！データサイエンスで重要な50の理論_Ｌ．ヴィッタート編.pdf`
（171ページ・ScanSnap系OCR・見開き構成の図解実用書）

## 1. 目的とスコープ

jisui2epub は縦書き小説特化で作られており、横書き本のスキャンPDFは想定外である。
本設計は横書き書籍を最低限変換できるようにするための仕様と実装方針を定める。

### 対象（Phase 1）

- 横書き書籍（エッセイ・実用書・児童書など）。日本語主体
- **左端揃えのテンプレート型段組（2〜3段）**。実本検証の結果、横書き実用書は
  段組が標準であり（§3.5: テキストページ103中89が2〜3段）、単段限定では
  実用にならないため Phase 1 に昇格した。続き物の段流し（用語集の左段→右段）と
  独立記事の並置（サイドバー）の両方を「帯を左→右の順に直列化」で扱う
- 柱・ノンブルの除去、段落復元、章見出し検出、ルビ（親文字の上に付く横ルビ）
- 挿絵・画像ページ検出（既存機構をそのまま利用。図解本では全面図版ページが
  約半数を占め、この経路が主力になる）
- 横書きリフローePub3 出力（`build_epub(horizontal=True)` — 生成側は実装済み）

### 対象外（Phase 2 以降に送る）

- 左端が揃わない複雑レイアウト（雑誌型の回り込み・セル状の図解・表・脚注）。
  テキスト行が僅少なページは画像ページ化に任せる
- 欧文の行末ハイフネーション結合（`inter-national` → `international`）
- vision_reocr.py / docai_reocr.py の横書き書き戻し（列スナップ・X降順ソート等が
  縦書き前提。横書き本の再OCRは Phase 2。検証サンプルのOCR品質は本文段で
  実用レベルだった＝§3.5）
- 縦横混植（縦書き本文中の横書き引用ブロック等）
- 書字方向の自動判別による自動切り替え（Phase 1 は明示フラグ。§7参照）

| 向き判定 | `span_is_vertical` | あり（縦積みアスペクト比） | 転置後はそのまま流用可 |

ePub側の既存対応の内訳（コード確認済み）:

- `_body_lines_to_xhtml(horizontal=True)` — 縦中横（tcy）の自動付与をスキップ
- `_make_epub_css` — `.hltr { writing-mode: horizontal-tb }` クラスあり
- `_make_opf(horizontal=True)` — `page-progression-direction="ltr"`、
  `primary-writing-mode` メタ抑止、表紙見開き指定の切替
- ただし **jisui2epub.py の `main`/`run_from_text` はこの引数を一切渡していない**
  （常に縦書きePub）。配線が必要（§8）。

## 3. 設計方針: 座標転置方式の採用

### 3.1 方式比較

**案A: 転置方式（採用）** — 横書きモード時、スパン収集直後に全bboxを90度転置し、
以降は既存の縦書きパイプラインを無改造で通す。

**案B: 並行パイプライン** — `analyze_page_h` / `assemble_text_h` 等を新設。

案Bを退ける理由: 現行コードの価値は CLAUDE.md に蓄積された検証済みヒューリス
ティック（ルビ紐付けのbisect方式・シフト・トゥ・フィット・改ページ判定・字下げ判定・
見出し重複排除…）そのものであり、横書き版を二重実装すると将来の修正が必ず乖離
する。転置方式なら既存ロジックの検証実績をそのまま引き継げ、縦書き経路への
影響もゼロにできる（転置は横書きモード時のみ発動）。

### 3.2 転置変換の定義

ページ高さを H として、点 (x, y) → (H − y, x)。bbox は:

```
(x0, y0, x1, y1) → (H − y1, x0, H − y0, x1)
```

ページ寸法は width′ = H、height′ = W に交換する。全座標は正のまま。

この変換で横書きの読み順・付属要素が縦書きの幾何に正確に写像される:

| 横書きの概念 | 転置後 | 縦書きパイプラインでの解釈 |
|---|---|---|
| 上の行ほど先に読む | x′ が大きい（右） | 右の列ほど先に読む ✓ |
| 行内は左→右 | y′ が小さい→大きい | 列内は上→下 ✓ |
| ルビは親の**上** | 親列の**右** | ルビは親行の右側（`attach_rubies` の前提） ✓ |
| 段落頭の1字下げ（左端から） | 列の上端が下がる | `starts_indented`（y0 > body_top + 0.55字）✓ |
| 行が右端まで届かず終わる | 列が下端まで届かない | `ends_short`（段落末尾判定）✓ |
| ページ下部が空く（章末） | 最左列が左端に届かない | 改ページ判定（`page_min_x` / `left_edge`）✓ |
| 見出し＝大きい活字の短い行 | 大きい活字の短い列 | `is_big`（サイズ比1.18+ ＋ 字下げ）✓ |
| 1行が複数スパン | 1列が複数スパン | x中心クラスタリング ✓ |
| 複数字スパンは横長 | 転置後は縦長 n:1 | `span_is_vertical` ✓ |
| **段組（段が左右に並ぶ）** | **帯が y′ 方向に並ぶ** | **対応する概念がない → §6 で新設** |

検証メモ（コードを読んで確認した対応）:

- `attach_rubies` の `gap = run.x0 - vl.x1`（ルビ左端−親列右端）は、転置後
  「行の上端−ルビ下端」＝横ルビと行の縦ギャップになり、`body_size*1.2` 上限も
  意味を保つ。Y→文字indexのbisect変換は転置後「X→文字index」となり、
  文字送りの実測ベース化（cell_height＝隣接差中央値）もそのまま効く。
- ルビのグルーピング（3pt格子のx中心）は転置後y中心格子となり、同一行上の
  横ルビを正しく1本化・ギャップ分割する。
- `analyze_page` の「孤立1文字列→横書き断片へ再分類」は、転置後は
  「孤立1文字行→hlines」となる。hlines は `classify_marginals` で柱・見出し
  候補として処理されるため実害はない。
- 青空文庫形式テキスト自体は書字方向に中立。横書きePubでは縦中横を
  出力しない（既存の `horizontal` 分岐）。

### 3.3 転置で意味が保存されない箇所（要対応・2点）

1. **柱・ノンブルの位置判定**。縦書き本の柱・ノンブルは本文（縦行）と直交する
   横書きでページ天地のマージンに出るため「本文領域（y上下端）の外」で捕捉
   できるが、横書き本の柱・ノンブルは本文と**平行**にページ天地へ出る。転置後は
   「本文列群と平行な、右端／左端の列」となり、そのy′範囲（＝元のx範囲）は
   本文と重なるので既存のyベース領域外判定に一切かからない → §5 で対応。
2. **段組**。縦書き小説に段組は事実上存在しないためパイプラインに対応する
   概念がなく、転置後の帯（y′方向に並ぶ段）は読み順ソートで行単位に交互混入
   する → §6 で対応。

### 3.4 スパイク検証①: 合成PDF（2026-07-20 実施済み）

本体無改造のまま `collect_spans` をモンキーパッチで転置し（
`scratchpad/test_yoko_transpose.py`）、合成横書きPDF（
`scratchpad/make_yoko_sample.py` — 2章・柱・ノンブル・横ルビ・1字下げ段落・
大活字見出し入り・単段組）を既存パイプラインに通した。

**転置だけで正しく動くことを実測確認:**

- 段落復元・読み順・折り返し行の結合・1字下げによる段落分割 — 完全に正しい
- 横ルビ3件（吾輩《わがはい》・掌《てのひら》×2）— すべて正しい親文字に付く。
  かな4字が漢字1字より物理的に広いケース（てのひら）も、既存の端トリムが
  範囲「の掌の」→「掌」と正しく絞った
- 章見出し2件とも `is_big` 経路で検出、章間の［＃改ページ］も正しく発行

**§3.3-1 の予測どおり漏れることを実測確認（＝§5 の必要性の裏付け）:**

- 柱「こころの散歩道」が毎ページ本文先頭の段落として混入
- ノンブルは1文字なら孤立文字→hline経路で偶然dropされるが、2文字以上は
  本文行と平行な列として混入（「10」が末尾段落化）

**合成PDF作成時の注意（再発防止）:** 字下げの全角空白を字形として
`insert_text` に含めると、スパンbboxに空白幅が乗る一方 `collect_spans` が
テキスト側だけ空白を剥がすため、セル境界が1文字ドリフトしてルビが隣の
漢字に付く**偽の症状**が出る。実スキャンのOCR層は字下げ空白を字形として
持たない（bboxは可視文字始まり。縦書き10冊の字下げ判定 `v.y0` が機能して
いる通り）ので、ジェネレータは空白分をペン位置オフセットで表現すること。

### 3.5 スパイク検証②: 実本サンプル（2026-07-20 実施済み・設計改訂の根拠）

「30秒でわかる！データサイエンス」（403×590pt・171ページ）で同じ転置スパイクを
実施。この本は見開き単位のテンプレート（左ページ＝記事、右ページ＝全面図版）で
構成される図解実用書であり、横書き実用書の代表的な難しさを一通り含む。

**ページ構成の実測**（本文級スパンのx区間クラスタによる帯数分布）:

| 帯数 | ページ数 | 内訳 |
|---|---|---|
| ほぼ無テキスト | 68 | 全面図版（見開き右）・口絵 → 既存の画像ページ検出が受け持つ |
| 1段 | 13 | 前付け・序文・はじめに |
| 2段 | 59 | 用語解説（続き物の段流し）・索引など |
| 3段 | 30 | 記事ページ（左サイドバー＋中央本文＋右サイドバー）＋章目次 |
| 5段 | 1 | 前付けの変則ページ |

**動いたもの:**

- テキスト層の品質は本文段（12pt）では実用レベル（ScanSnap相応の誤字はある:
  第2草・3()秒 等）。`detect_body_size`＝12.2pt と安定（文字数重み付けにより
  サイドバー9.5ptでなく本文が勝つ）
- サイドバー本文（9.5pt＝本文比0.78）はルビ閾値（0.68）と見出し閾値（1.18）の
  間に収まり本文扱いになる ✓
- 段が混ざらない範囲では折り返し行の段落結合が機能
- 記事見出し（15.6pt＝本文比1.28）は `is_big` の対象になる

**破綻したもの（すべて多段レイアウト起因）:**

1. **段の行単位交互混入**: 隣り合う段の行が読み順ソート（転置後x′降順＝元の
   y昇順）で交互に混ざり、本文が完全に壊れる。例:「アルゴリズムは同じことを
   人間がやるお手本を処理することで意思決定のやり方**ギャラッブ世論調査**
   を学ぶ…」→ §6 の帯分割で解決する（最重要）
2. **柱・ノンブル混入**: この本は柱（第2章 不確実性・9pt）とノンブルがページ
   **上端**にあり、y′ベースの既存判定では捕捉不能 → §5 のx′バンド拡張で解決
   する設計のまま正しい（柱9pt < 本文×0.8 のためバンド推定を汚染しない
   ことも確認）
3. **サイドバーの極小断片がルビとして隣の段に付く**: 相互参照「(p.4)」等の
   6pt級スパンがルビ判定（≤8.3pt）に食い込み、「出《ご》力値」のような誤付着を
   起こす → §6 の帯分割で親候補が帯内に限定され大半が解消。実用書では
   `--ruby drop` の運用も推奨（§7）
4. **OCRサイズ揺れによる見出し誤検出**: 本文12.5pt行が15pt級に誤認され
   「学生で有意な差が見つかれば,帰無仮説」が中見出し化 → 既知リスクとして
   残す（縦書きでも同種のリスクは `is_big` の複合条件で抑えている。帯分割後は
   帯内の字下げ条件が正しく効くため発生頻度は下がる見込み。目次精錬も防波堤）
5. **見出しの飾り記号混入**: 装飾枠が「口標本抽出□」のようにOCRされる →
   Phase 1 では許容（テキスト校正で直す想定。`normalize_ocr_text` への
   追加は誤爆リスクと相談して Phase 2）

## 4. 変更箇所一覧

方針: 既存関数にキーワード引数（既定値＝現行動作）を足す形で拡張し、
**縦書き経路のバイト単位の出力不変**を保証する。`analyze_page` と
`detect_body_size` は vision_reocr.py から import されているため、
シグネチャは後方互換必須。

| 箇所 | 変更内容 |
|---|---|
| 定数部 | `transpose_spans(spans, page_h)` を新設（bbox転置のみ、5行程度） |
| `analyze_page(fpage, num, body_size, horizontal=False)` | 冒頭で `collect_spans` の結果を転置し、`Page(num, H, W)` と寸法交換。以降は無改造 |
| `classify_marginals(pages, body_size, horizontal=False)` | §5 の x′バンド拡張（横書きモードのみ発動） |
| `split_columns(pages, body_size)`（新設） | §6 の帯分割。横書きモードのみ呼ぶ。Page列を「帯＝仮想ページ」列に展開 |
| `Page` | 帯の本文範囲 `band_top` / `band_bottom`（省略時None＝従来動作）と `no_break`（帯間の改ページ抑止）を追加 |
| `assemble_text` | ページ局所の本文範囲（`band_top/band_bottom`）があれば優先、`no_break` ページでは意図的改ページ判定を抑止（§6.3。既定値では従来と完全同値） |
| `inspect_page(doc, num, body_size, horizontal=False)` | 転置後座標＋帯分割結果を表示。ヘッダに「横書きモード（転置座標: x′=H−y）」の注記 |
| `main` | `--horizontal` フラグ追加、各関数へ配線。§7 の自動判別警告 |
| `run_from_text` | `--horizontal` を `build_epub(horizontal=...)` へ配線 |
| `build_epub` ほか ePub 生成部 | 変更なし（既存の `horizontal` 引数を使うだけ） |

推定差分規模: +200〜280行程度（§5 が約50行、§6 が約120行）。

## 5. 柱・ノンブル検出の x′バンド拡張（新規仕様）

`classify_marginals` に横書きモード限定の処理を追加する。転置座標系で記述する
（x′ 右端＝元のページ上端、x′ 左端＝元のページ下端）。

### 5.1 本文バンド（x′ 方向）の推定

既存の body_top/body_bottom 推定（y方向）と対称の手法:

- 各ページで「セル数 ≥ 3 かつ size ≥ body_size × 0.8」の列（＝本文らしい行）の
  x′中心の最大・最小を取り、全ページの中央値を body_right′ / body_left′ とする。
- 柱・ノンブルは本文よりフォントが小さい（CLAUDE.md 既知知見。検証サンプルでも
  柱9pt/本文12.2pt＝0.74）ため、サイズフィルタで大半は推定から除外され、
  中央値汚染を防ぐ。本文同サイズの柱を持つ本はバンドを汚染しうるが、その場合も
  既存のテキスト頻度照合（`norm_hashira` + `HASHIRA_MIN_FREQ`）が後段で効く
  ため、Phase 1 では既知の制限とする。

### 5.2 候補収集と判定

- 既存の候補収集条件「y範囲が本文領域外の縦行」に、横書きモード時は
  「**x′中心が [body_left′ − cell×0.5, body_right′ + cell×0.5] の外**の列」を追加。
- `in_margin` の定義を「y中心が領域外 **or**（横書きモード時）x′中心がバンド外」
  に拡張。
- 判定ルール自体（NOMBRE_RE→ノンブル drop、`is_junk_line`→drop、サイズ大→見出し、
  頻度≥3→柱、マージン内×頻度≥2→柱、マージン内×小サイズ→柱…）は**一切変えない**。
  座標軸の読み替えだけで既存ルールを再利用する。
- 領域推定・判定とも既存同様「中心座標」で行う（端座標だと数ptで判定漏れする
  既知知見に従う）。
- 検証サンプルへの適用見込み: 柱「第N章 …」は章ごとに3ページ以上出るため
  頻度照合で drop、上端隅のノンブルは NOMBRE_RE で drop。どちらも本文行の
  開始（y≈76〜）よりページ上端側（y≈40）にあり x′バンド外になることを
  スパン実測で確認済み。

### 5.3 やらないこと

- ルビ誤分類の柱移送（既存の y ベース処理）の x′拡張は行わない。小フォントの柱が
  ルビ扱いのまま残っても、`attach_rubies` のギャップ上限（body_size×1.2）と
  かな比率チェックで親行に付かず、未対応ルビは出力に乗らないため実害がない。
- 本文列端のノンブル剥がし（`_nombre_junk_char`）は既存のまま。剥がしは
  バンド外にはみ出したセル限定なので、横書き本の行末の正当なASCIIには
  誤爆しない。

## 6. 段組（帯分割）対応（新規仕様・実本検証で Phase 1 に昇格）

### 6.1 位置づけとデータフロー

`classify_marginals`（柱・ノンブル除去後）と `assemble_text` の間に
`split_columns` を新設する。転置空間で「段」は y′ 方向に並ぶ帯になる。
各ページを帯ごとの**仮想ページ**列に展開し、帯を y′ 昇順（＝紙面の左→右）に
並べて `assemble_text` に渡す。これにより:

- 続き物の段流し（用語集）: 左段の全行 → 右段の全行 の順になり、段末で切れた
  段落は「段末行が行末まで達している（ends_short=False）＋次帯先頭が字下げなし」
  の既存ロジックで自然に結合される
- 独立記事の並置（サイドバー型）: 左サイドバー → 中央本文 → 右サイドバー の
  ブロック順で直列化される（紙面の読み順の慣行どおり）

### 6.2 帯の検出アルゴリズム（実本で検証済み）

単純な「x区間の和集合の隙間」では、OCRのbboxが溝に1〜2ptはみ出しただけで
帯が融合する（検証サンプルp42で実測: 左帯末尾行 x=33-114 が中央帯 x0=115 に
接触して融合）。**横書きは左揃えが強い信号**なので、行の左端で帯を作る:

1. ページ内の本文級の行（size ≥ body×0.75 かつ 4文字以上）の**左端**（転置後は
   y0′）を昇順に走査し、間隔 ≤ body×1.8（字下げ1字＋α）で連結クラスタ化する
2. 3行以上が属するクラスタだけを帯の種とする（センタリング見出し・引用ブロック・
   ノイズを種にしない）
3. 帯の範囲＝[クラスタ左端, 所属行の右端の中央値]。**範囲が重なる帯同士は統合**
   する（中央帯内の字下げ引用ブロックが別帯になるのを吸収。p42/p46 で実測確認）
4. すべての行・ルビ・hlineを「重なり最大」の帯に割り当てる（センタリング見出し
   は左端クラスタに入らないが、この割当で正しい帯に落ちる）。どの帯とも
   重ならない要素は最近傍の帯へ
5. 帯が1つならページをそのまま返す（単段ページ・縦書きモードと同一挙動）

実測: 検証サンプルの代表ページで 2段（用語解説）・3段（記事）・1段（序文）を
正しく分離。帯数上限は4（それ以上は変則ページ＝そのまま1段扱いにフォールバック
し、壊れた読み順よりページ順を優先する）。

### 6.3 仮想ページの本文範囲と改ページ抑止（assemble_text の小改修）

- 帯ごとの仮想 `Page` に `band_top` / `band_bottom`（転置後y＝帯の左右端）を
  持たせる。`assemble_text` の字下げ判定（`starts_indented`）と段落末尾判定
  （`ends_short`）はページ局所値があればそれを使う。
  - 補足: 字下げ側は既存の `eff_top`（ページ実測topへの追従）で大半動くが、
    `ends_short` はグローバル body_bottom 比較のため、幅の狭いサイドバー帯では
    全行が「途中終わり」＝1行1段落に壊れる。band_bottom が必須なのはこのため
- 同一ページ内の帯間では意図的改ページ判定（`prev_page_short`）を発動させない
  （最終帯のみ判定に参加）。サイドバー帯は下端まで達しないのが常態であり、
  帯ごとに判定すると偽の［＃改ページ］が量産されるため
- ルビは帯に割り当てた上で仮想ページに載せるため、`attach_rubies` の親候補が
  自動的に帯内へ限定される（§3.5 の誤付着対策を兼ねる）

### 6.4 縦書きモードへの影響

`split_columns` は横書きモード限定で呼ぶ。`assemble_text` の拡張は
`band_top/band_bottom` 未設定（None）時に従来式へ完全に一致させる。
縦書き本の上下2段組は稀（Phase 1 対象外のまま）。

## 7. CLI 仕様

```bash
# 横書き本の変換（テキスト＋ePub）
.venv/bin/python jisui2epub.py input.pdf --horizontal --epub
# 図解実用書などルビが実質ない本はルビ除去を推奨（誤検出ルビの混入防止）
.venv/bin/python jisui2epub.py input.pdf --horizontal --ruby drop --epub
# 校正済みテキストから横書きePub再生成
.venv/bin/python jisui2epub.py input.pdf --from-text 校正済み.txt --horizontal
# 横書きモードのページ構造デバッグ（帯分割結果も表示）
.venv/bin/python jisui2epub.py input.pdf --horizontal --inspect 42
```

- `--horizontal`: 解析を転置モードで行い、ePubを横書き（hltr / ltr）で生成する。
  既定は現行どおり縦書き。
- **自動判別は Phase 1 では「警告のみ」**: `detect_body_size` のサンプリングと同じ
  ページ群で、複数文字スパンの `span_is_vertical` 判定を文字数重みで集計し、
  横書き優勢（横 > 縦×2 目安）なのに `--horizontal` 未指定なら
  「横書きの可能性があります。--horizontal を検討してください」と警告する
  （逆も同様）。自動切替にしない理由: 1文字1スパンの旧OCRでは向き票が
  ほぼ得られず判定不能になるため、誤判定で全損するより明示指定が安全。
  判別精度の実績が積めたら Phase 2 で `--writing-mode auto` を既定化する。

## 8. ePub 生成の配線

## 10. 既知の制限（Phase 1 時点で明記するもの）

- 帯検出は「左揃え・3行以上・帯数4まで」のテンプレート型のみ。左端が揃わない
  誌面（雑誌型回り込み・セル状図解）は1段フォールバック＝読み順が乱れる可能性
- 独立記事並置（サイドバー）で、前帯最終行がちょうど行末まで達しかつ次帯先頭が
  字下げなしの場合、別記事の段落が結合されることがある（構造上不可分。校正対応）
- 本文と同サイズ・同フォントの柱はバンド推定を汚染しうる（§5.1）
- サイドバー等の小さめ本文が body×0.68 を割る本では、段丸ごとルビ誤分類の
  リスクがある（検証サンプルは0.78で安全圏。発生したら閾値でなく帯ごとの
  サイズ再評価を Phase 2 で検討）
- OCRサイズ揺れによる見出し誤検出・飾り記号の見出し混入は残存（§3.5-4,5）
- 欧文はハイフネーション結合なし・比例幅による字下げ判定の精度低下がありうる
- 表・図解・脚注は構造化しない（画像ページ化されるか、乱れたテキストになる）
- vision_reocr.py / docai_reocr.py は縦書き専用のまま（横書き本はスキャナ内蔵
  OCRのテキスト層をそのまま使う）

## 12. 実装結果（2026-07-20 実装完了）

§11 の全ステップを実装し検証した。設計からの主な変更点:

1. **帯検出は `split_columns` でなく `analyze_page` のスパン段階に前倒し**
   （`_detect_span_bands`）。実装して判明: 段間の溝（1字幅≈10pt）は列クラスタの
   Yギャップ許容（本文×1.6≈20pt）より狭いため、vline化の後に帯を検出しても
   同じベースライン上の別段の行が既に1本に融合している（30秒DS p42で実測:
   「検討したい母集団すべ**人は計測値の収集に**…**●関連参照項目**」）。
   スパンに帯indexを付与し `find_col` の統合を帯内に制限、ルビの
   グルーピングも帯キーで仕切った。`split_columns` は展開のみを担う
2. **見出しの飾り囲みは「許容」でなく「剥がし」が必須だった**（§3.5-5 の
   想定を変更）。囲み枠は 口/ロ/□/■ にOCR化けし、かな無しの理論名見出し
   （口標本抽出□）が `is_junk_line` の「かな無し＋記号」規則で **3箇所**
   （classify_marginals のdrop・assemble_text の行ノイズ除去・見出し有効性
   判定）すべてから排除され、見出しが完全消失した。`_strip_heading_ornaments`
   を新設し、横書きモードでは3箇所とも剥がし後のテキストで判定する。
   併せて横書き限定の緩和・強化: 純漢字2〜8字見出しの許可（標本抽出・
   平均回帰）、hline見出しの不採用（紙面の垂直テキスト＝図版ノイズ）、
   ノイズ署名 `_H_NOISE_RE`（一一連続・《》・℃・行頭ー）による拒否、
   テキスト僅少ページの2〜3字大活字の見出し化免除の廃止
3. CLI は設計どおり（`--horizontal`・自動判別は警告のみ・`--from-text` 連携）
