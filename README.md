# jisui2epub

自炊PDF（OCRテキスト層付き）から青空文庫形式テキストとリフロー型ePub3を生成するツール。

紙の本を裁断・スキャン（ScanSnap等）して自動OCRしたPDFから本文を抽出し、
手作業で行っていた以下の校正作業を自動化する:

- **天柱・地柱の除去** — ページ上下に毎ページ繰り返されるタイトル・章名
- **ノンブル（ページ番号）の除去**
- **ふりがな→ルビ変換** — 座標とフォントサイズからルビを検出し `親文字《るび》` 形式に
- **段落の復元** — 字下げ位置から段落境界を判定（会話文の天付きにも対応）
- **章見出しの検出** — ［＃改ページ］＋［＃「…」は中見出し］マーカー付与（ePubの目次になる）
- **挿絵OCRノイズの除去** — 挿絵ページから出る記号・珍漢字の羅列を除去
- **OCR系統誤りの後処理正規化** — 文脈から機械的に正せる誤りを自動修正
  （ｌ/Ｉ→ダッシュ――・長音ー、○→数字文脈で〇、〃″→〝〟、拗音の小書きカナ並字化）
- **意図的な改ページの反映** — 章末・章扉・目次・詩など本文が途中で終わるページを
  検出して ［＃改ページ］ を付与。ePubでは目次項目を増やさずに改ページとして描画
- **挿絵・口絵・章頭ページの画像ページ化** — 本文行がほぼ無くインクの多いページを
  検出してJPEG化（余白自動トリム）。テキストには ［＃「挿絵」の図（…）入る］ タグと
  `タイトル_images/` フォルダ、ePubには公式電子書籍と同じ全画面の画像ページとして
  該当位置に綴じ込む（目次は増えない）

## 使い方

```bash
# セットアップ（初回のみ）
python3 -m venv .venv
.venv/bin/pip install pymupdf

# PDF → 青空文庫形式テキスト
# ファイル名が「タイトル_著者名.pdf」なら --title/--author は省略可（自動取得）
.venv/bin/python jisui2epub.py タイトル_著者名.pdf

# ePubまで一気に生成（生成機能は内蔵。表紙は既定でPDFの1ページ目）
.venv/bin/python jisui2epub.py 本.pdf --title "タイトル" --author "著者名" --epub

# 出力テキストをエディタで校正したあと、校正済みテキスト＋元PDFからePubを再生成
# （表紙と挿絵画像はPDFから取得。--epub 指定は不要）
.venv/bin/python jisui2epub.py 本.pdf --from-text 校正済み.txt

# 前付け・後付け（表紙・目次・奥付）を除外して本文ページだけ変換
.venv/bin/python jisui2epub.py 本.pdf --pages 10-360

# ページ構造のデバッグ表示（1始まりのページ番号）
.venv/bin/python jisui2epub.py 本.pdf --inspect 20,21
```

主なオプション:

| オプション | 説明 |
|---|---|
| `--title` / `--author` | タイトル・著者名。省略時は「タイトル_著者名.pdf」形式のファイル名から自動取得（`_` 優先、次いで `-` で分割） |
| `--from-text FILE` | 校正済みの青空文庫形式テキストからePubを再生成。挿絵はテキスト隣の `〜_images/` があればそれを、なければPDFから再取得 |
| `-o FILE` | 出力テキストのパス（既定: PDFと同じ場所に `タイトル_著者名.txt`） |
| `--pages 10-360` | 対象ページ範囲。`5,8,10-` のような指定も可 |
| `--ruby drop` | ルビを除去（古いOCRでルビ精度が悪い場合） |
| `--no-indent` | 段落頭の全角空白を入れない |
| `--no-ocr-fix` | OCR系統誤りの後処理正規化を行わない |
| `--no-images` | 挿絵・口絵・章頭ページの画像ページ化を行わない |
| `--epub` | リフロー型縦書きePub3まで生成（生成機能は内蔵） |
| `--cover-page N` | ePub表紙にするPDFページ（既定=1ページ目、`0`で表紙を自動生成） |
| `--cover-image FILE` | 表紙に任意の画像（JPEG/PNG）を使う |
| `--inspect N` | ページ解析結果を表示して終了（チューニング用） |

## 再OCR前処理（vision_reocr.py / docai_reocr.py）— 古いOCRの精度を底上げ

スキャナ内蔵OCR（特に古い機種）は文字化けが多く、jisui2epub.py の抽出精度の
上限になっている。`vision_reocr.py` は Google Cloud Vision API でPDFを再OCRし、
より高精度な透明テキスト層に差し替える前処理ツール（jisui2epub.py本体は無改造）。

```bash
# セットアップ（初回のみ）
.venv/bin/pip install google-cloud-vision

# Google Cloudの準備:
#   1. GCPプロジェクトを作成し、Vision APIを有効化
#   2. 課金を有効化（少数ページのテストなら月1000ユニットの無料枠に収まる）
#   3. サービスアカウントキー（JSON）を発行
export GOOGLE_APPLICATION_CREDENTIALS="/path/to/key.json"

# 再OCR実行（既定出力は <入力>_vision.pdf）
.venv/bin/python vision_reocr.py 本.pdf

# 途中で中断した場合、続きのページから再開できる
.venv/bin/python vision_reocr.py 本.pdf --start 150

# 数ページだけ試す
.venv/bin/python vision_reocr.py 本.pdf --start 10 --end 20 -o test.pdf

# 再OCR後のPDFをjisui2epub.pyにかける
.venv/bin/python jisui2epub.py 本_vision.pdf --title "タイトル" --author "著者名" --epub
```

本文とルビは異なるフォントサイズ（本文の約半分）で書き戻されるため、
jisui2epub.py本体のルビ検出・柱ノンブル除去はそのまま機能する。

ルビの親文字への対応付けは、正解テキストとのルビペア単位の実測で
74〜91%（ルビ密度が非常に高い児童書・新書4冊で検証。詳細はCLAUDE.md）。
残る誤りの主因はVision API自体の極小ルビの読み取り精度で、
出力テキストの校正時にルビを重点的に確認するとよい。

### さらに高精度を求める場合（docai_reocr.py）

`docai_reocr.py` は同じ仕組みのOCRエンジンを Google Cloud **Document AI**
（Enterprise Document OCR）に差し替えた版。**無料枠がなく$1.50/1000ページ**
（300ページの本1冊≈70円）かかるが、実測で本文の誤読がVisionより少なく
（地下室からのふしぎな旅10ページの本文でVision誤り42字に対しDocAI30字・
うち実質的な誤読は1字のみ。残りは字種の揺れとノンブル混入）、Visionが
検出漏れする1文字だけの極小ルビも自前で検出できる。処理速度は約2秒/ページ
（Visionの約4倍遅い）。

```bash
# セットアップ（初回のみ）
.venv/bin/pip install google-cloud-documentai
# Google Cloud ConsoleでDocument AI APIを有効化（課金必須）した上で、
# OCRプロセッサを作成する:
.venv/bin/python docai_reocr.py --create-processor

# 使い方はvision_reocr.pyと同じ（既定出力は <入力>_docai.pdf）
.venv/bin/python docai_reocr.py 本.pdf
.venv/bin/python docai_reocr.py 本.pdf --start 10 --end 20 -o test.pdf
```

まずは無料のvision_reocr.pyを試し、誤字の多さが気になる本にdocai_reocr.pyを
使う、という使い分けを推奨。

## スキャン画質は「スーパーファイン」推奨

同一の裁断済紙本『書を捨てよ、町へ出よう』（角川文庫・320ページ）を ScanSnap の
画質ノーマルとスーパーファインでスキャンし、公式ePub由来のテキストと
文字単位で突き合わせた実測結果:

| 画質 | 文字一致率 | 誤字数（本文約12.9万字中） | PDFサイズ |
|---|---|---|---|
| ノーマル（150dpi） | 95.91% | 5,253字 | 17MB |
| スーパーファイン（300dpi） | **97.65%** | **3,016字（約43%減）** | 39MB |

「い→し」「ン→ソ」「本→木」のような字形混同はスーパーファインでほぼ消える。
ファイルサイズは2.3倍になるが、OCR誤字の校正の手間が大きく減るため
**スーパーファインでのスキャンを推奨**。なお小書き仮名（ツ↔ッ）・半濁点（ボ↔ポ）・
ダッシュ（──→ｌ）の誤りは解像度を上げても残る（OCRエンジン側の限界）。

このうち文脈から機械的に正せるものは内蔵の後処理正規化が自動修正する
（この本では約350〜400箇所/冊）。正規化込みの誤字数はノーマル4,924字・
スーパーファイン**2,634字**まで減る。

## 仕組み

PyMuPDFでOCRテキスト層をスパン（文字＋座標＋フォントサイズ）単位で取得し、

1. 本文フォントサイズを推定（文字数重み付き最頻値）
2. スパンをx座標で縦行にクラスタリング（縦書き前提、右→左）
3. 本文サイズの約半分のスパン＝ルビ候補。y範囲を親行の文字セルに写像して漢字連続にスナップ
4. 本文領域（縦行のy範囲の中央値）の外にある行＝柱・ノンブル候補。
   数字のみ→ノンブル、数字を除いた同一テキストが3ページ以上→柱、として除去
5. 章見出しは「本文より大きいフォント」または「柱と同じテキストがページ右端に深い字下げで現れる行」
   （柱テキスト＝章名の対応を利用。OCR揺れはdifflibのあいまい一致で吸収）
6. 段落は行頭の字下げ量と直前行の短さから復元

## 実測値（temp_sample の2冊）

- ほんものの魔法使（創元推理文庫・2026年スキャン）: 第1章の文字一致率 **98.1%**、
  全21章の見出しを検出、ルビ208箇所
- グリックの冒険（岩波少年文庫・2013年スキャン）: 全章検出、段落構造は手作業校正版と一致。
  古いOCRはルビ座標が不正確なため、ルビの親文字境界に誤りが出る（`--ruby drop` も検討）

## 限界（残る手作業）

- **OCRの誤認識文字そのもの**（例: フラック→ブラック、レースカバー→レースカゞ〈ｌ）は直せない。
  エディタでの一括置換・校正は引き続き必要（元作業の2〜3割程度）。
  校正後は `--from-text 校正済み.txt` でePubを再生成できる。
  なおルビ内に漢字がまじる誤認識（女《おん姪》等）は、同じ親文字への正常な
  ルビの本内多数決＋類似度で自動訂正される（実測: 霧のむこうのふしぎな町で
  69箇所、誤訂正はいずれも読みの揺れレベル）
- 表紙・帯・奥付・目次ページはノイズになりやすい（表紙と画像化されたページは
  自動処理されるが、広告など文字の多いページは残る）→ `--pages` で除外するのが手っ取り早い
- 挿絵ページは画像ページ化されるが、本文行が多いと判定されたページのかな入りノイズは残ることがある
- 傍点・圏点・字下げブロックなどの組版情報は検出できない
- 横書きの本は未対応（縦書き小説向け）
- 古いOCR（〜2010年代前半のScanSnap）はルビ精度が低い。`vision_reocr.py` で
  Google Cloud Vision APIによる再OCR後に変換すると精度が上がる
