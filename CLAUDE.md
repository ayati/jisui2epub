# CLAUDE.md

## jisui2epub

自炊PDF（OCRテキスト層付き縦書き小説）→ 青空文庫形式テキスト → リフロー型ePub3 変換ツール。
単一ファイル `jisui2epub.py`、依存は PyMuPDF のみ（`.venv` にインストール済み）。

```bash
# 実行（ファイル名が「タイトル_著者名.pdf」なら --title/--author 省略可）
.venv/bin/python jisui2epub.py input.pdf --title T --author A [--epub]
# 校正済みテキスト＋元PDFからePub再生成（表紙・挿絵はPDFから取得）
.venv/bin/python jisui2epub.py input.pdf --from-text 校正済み.txt
# 構文チェック
.venv/bin/python -m py_compile jisui2epub.py
# ページ構造デバッグ（ヒューリスティック調整時に必須）
.venv/bin/python jisui2epub.py input.pdf --inspect <1始まりページ番号>
```

## アーキテクチャ

パイプライン: `analyze_page`（スパン→縦行/横行/ルビ分類）→ `classify_marginals`
（全ページ横断で柱・ノンブル・挿絵ノイズ特定）→ `classify_image_pages`＋
`render_image_page`（挿絵・口絵・章頭の画像ページ化）→ `attach_rubies`
（ルビ→親文字対応）→ `assemble_text`（段落復元・見出し検出・青空文庫形式出力）
→ `parse_aozora_text`＋`build_epub`（内蔵、novel_downloader.py から移植）でePub化。

ePub生成部（`# ePub3生成` セクション、`_make_epub_css`〜`parse_aozora_text`）は
novel_downloader.py からの移植コード。表紙は既定でPDF 1ページ目をJPEGレンダリング
（`--cover-page N` で変更、`0` でタイトル文字の自動生成表紙、`--cover-image` で画像指定）。
Pillowなしでも動作（自動生成表紙のみSVGフォールバック）。移植時の変更点:
奥付の入力元表記・空URL行抑止、`build_epub` に `cover_image_data` 引数追加、
フォント検出の遅延実行化。本家と機能同期する場合は同名関数を差し替える。

重要な知見（temp_sample の2冊で検証済み）:

- **OCR世代でスパン構造が違う**: 古いScanSnap(2013)は1文字1スパン、新しい(2026)は1行1スパン。
  `analyze_page` はどちらもx中心クラスタリングで縦行に統合する
- **pypdfはCMap解釈に失敗して文字化けする**。PyMuPDF必須
- **柱・ノンブルは本文よりフォントが小さく、ルビ判定(0.68倍以下)に食い込む**ことがある
  → classify_marginals で本文領域外のルビをマージン行に移送してから判定
- **章見出しの2方式**: 岩波少年文庫型＝本文より大きい横書き行（サイズ比1.18+）。
  創元推理文庫型＝本文と同サイズの縦行だが柱と同一テキスト・ページ右端・深い字下げ(1.5字超)
- **柱のOCRは毎ページ揺れる**（靴棚→腱棚/腱場/農場）→ 数字除去＋difflibあいまい一致(0.66)で照合
- **重複見出し**: 章の扉ページと本文ページ両方に章題が出る本がある → 数字前方一致＋
  類似度0.75で15段落以内なら排除（(1)(2)(3)連続章を潰さないため数字は必須比較）
- **マージン判定は行のy中心で行う**。端座標だと数ptの差で判定漏れする
- **OCR後処理正規化(normalize_ocr_text)は文脈で確定できる誤りだけ直す**。
  小書き母音（ァィゥェォ）はオノマトペ（ゴォーッ・ワァーッ・キィーッ）で任意の音の
  後に正当に付くため変換禁止（グリック手作業版で誤修正13箇所を確認して除外した）。
  ヶも地名（関ヶ原）で正当。拗音ャュョのみ直前イ段チェックで安全に直せる。
  正文でカタカナ直後にダッシュが来るのは「ン――」「ム――」のみ（3冊で確認）
  → ン・ム以外のカタカナ+ｌ/Ｉ+かな は長音ーと判定してよい
- **文末の句点。→中黒・の誤認識は「位置」で確定できる**。正当な中黒はカタカナ語・
  人名・漢字語の区切りがほぼ全てで、閉じ括弧「」』）の直前・行末・。の直前に来る
  正当例は正解テキスト5冊・中黒約530個中0件（グリック手作業版にも見逃し3件が
  残っていた＝人手でも漏れる誤り）。ひらがな間など文中の中黒は正当例があるため
  触れない。実測: 霧117箇所・伯爵夫人19箇所、正解3冊への誤爆0
- **ルビ内の漢字混入は本内多数決で訂正できる**（fix_ruby_kanji）。同じ親文字への
  正常な（仮名のみ）ルビは本の中に多数出現する（女《おんな》×28 対 誤り10件）。
  かな部分の類似度上位0.25幅内で最頻の候補を採用（だいどぢ×1よりだいどころ×3を
  優先）。頻度1の候補は同親に頻度4以上があれば自身も破損疑いで除外（売《やつ》対策）。
  ゑ・ゐは現代作品のルビにほぼ出ず「み」の破損（すゑか）の方が多いので正常扱い
  しない。霧のむこうのふしぎな町でOK48/NG4、グリックでOK28/NG3を確認。NGは
  いずれも仮名への置換で置換前より悪化しない。正解ルビが本内に一度も正しく
  OCRされない語（住処→すゑか/すぶかのみ）はこの方式では直せない
- **--from-text の画像は pNNNN.jpg のページ番号でPDFから再レンダリング可能**。
  テキスト隣の `〜_images/` ディレクトリを優先し、なければPDFから再取得
  （両経路でバイト列一致を確認済み）。校正でファイル名を変えなければ再現できる
- **画像ページ検出は「本文行≤4 × インク率」の2軸**（タイム・リープ上下巻で較正）。
  インク率＝0.3倍縮小グレースケールの暗画素率(<200)。閾値は本文ページ中央値の
  0.7倍（最低0.018）。白ページ・「上巻完」等の余白ページは弾かれ、章頭画像
  （ink 0.048〜）も拾える。表紙ページ（--cover-page）は本文抽出ごと除外
- **画像ページ上の見出しは柱一致か単独章パターンのみ採用**（口絵・目次の飾り文字
  対策）。目次ページの章題は個別に柱一致してしまうため、`_demote_image_only_headings`
  で「本文8字以下＋画像のみのセクション」の見出しを後方類似見出しがあれば取り消す。
  **縦書き目次のOCRは読み順が崩れる**（あとがきがわりに→りに力きがわ）ため
  SequenceMatcherでなく文字集合オーバーラップ(0.75)でも比較する
- **ePubの画像ページはspine分割方式**: エピソード本文中の「改ページに挟まれた単独
  図タグ」を `_split_episode_images` で抜き出し専用XHTML（img0001.xhtml…）に。
  エピソード先頭ファイル名 ep{n:04d}.xhtml を維持することで toc/nav は無改修
- **意図的な改ページ検出は「ページ最左行の位置」で行う**（縦書きは右→左に詰まる）。
  本文左端＝全ページ最左行位置の下位25%点、行送り2.5本分以上空けば章末・扉・目次と
  みなし［＃改ページ］を発行。ただし**最終行がページ下端まで達している場合は段落
  継続中（挿絵で左側が空いただけ）なので改ページしない**（prev_line_short ガード。
  グリックの挿絵ページで文の途中に改ページが入る誤検出を潰した）。
  ePub側は見出しなしセクションを前エピソードに連結し div.pagebreak
  （break-before: page）で描画 → 目次項目を増やさない

## vision_reocr.py（前処理: Google Cloud Vision APIによる再OCR）

自炊PDFのスキャナ内蔵OCR（ScanSnap等）は文字化けが多く、jisui2epub.py の抽出精度の
上限になっている。本スクリプトはVision APIの `document_text_detection` で全ページを
再OCRし、透明テキスト層として書き戻す。jisui2epub.py 本体には一切手を入れず、
前処理として独立に使う（`analyze_page` 以降は無改造で動く設計）。

```bash
.venv/bin/pip install google-cloud-vision
export GOOGLE_APPLICATION_CREDENTIALS="/path/to/key.json"
# GCPコンソールで対象プロジェクトの Vision API 有効化＋課金有効化が必要
.venv/bin/python vision_reocr.py input.pdf                    # 全ページ、既定出力は <入力>_vision.pdf
.venv/bin/python vision_reocr.py input.pdf --start 100        # 中断からの再開（既存出力を土台にする）
.venv/bin/python vision_reocr.py input.pdf --start 10 --end 20 -o test.pdf  # 範囲指定テスト
# 再OCR後のPDFはjisui2epub.pyにそのままかける
.venv/bin/python jisui2epub.py input_vision.pdf --title T --author A --epub
```

設計の要点（temp_sample の2冊で検証済み）:

- ページの埋め込み画像（スキャン原本。複数あれば最大面積のもの）をそのままVisionに
  送る。ページ再レンダリングは解像度劣化＋余計な処理コストになるため使わない
- **word単位でのinsert_textboxは失敗する**: 縦書きは1文字ずつboxが積まれるため、
  横書き前提の行送りロジックでは複数文字の単語が列幅に収まらず無言でスキップされる
  （返り値が負のdeficitになる＝ほとんどの文字が消える）。文字（シンボル）単位で
  insert_text（原点指定・行送りなし）を使うことで解決した
- **ルビ判定は縦書きの列幅（rect.width）で行う**。句読点・小書き仮名（っ・ょ等）は
  本文と同じ列幅を保ったまま高さ（インク）だけ小さくなるため、高さで判定すると
  誤って除外してしまう（グリックの冒険P.11で実測: 高さ判定だと「フラック」の
  「っ」や読点が本文から消えた）。幅判定なら本文文字は保たれ、ルビ（列自体が
  本文の0.68倍未満）だけ正しく検出できる。半角英数字はルビに出現しないため
  判定対象から除く（数字は幅が狭く誤検出しやすいため）
- **ルビも書き戻す**（除外して捨てない）。本文は文字ごとの実インク高さでなく
  ページ横断の一律フォントサイズを使う（インク高さは句読点等で本文でも小さく
  なるため、個々の高さをそのままフォントサイズにすると後段のjisui2epub.py側
  `attach_rubies` の0.68倍判定を誤爆させる）。ルビはページごとのVision実測幅
  中央値で書き戻す（詳細は下記「ルビ密度が非常に高い本」の原因3）。
  **「縦行数・ルビ数が仕様通り」＝「親文字への紐付けが正しい」ではない**
  （`--inspect`の件数一致だけで正しいと誤認しないよう要注意。ペア単位の検証は
  `scratchpad/eval_ruby.py`で行う）
- **書き戻すフォントサイズは「ページ自身のインク幅」ではなく、`detect_body_size`
  （jisui2epub.py本体から直接import）でドキュメント全体から求めた基準値を使う**。
  ルビ/本文の判定（is_ruby）自体はページ自身のインク幅の中央値との相対比較で
  行うが（実際のルビ幅は本文のほぼ半分なので頑健）、実際に書き戻す値は別に
  固定する。理由: Visionのインク幅（文字の可視範囲）は、フォント設計上の
  字面外マージンを含む「フォントサイズ」そのものより一回り小さく出る（実測で
  本文インク幅／旧OCR本文フォントサイズ＝0.66〜0.71倍と、ちょうどルビ判定の
  0.68倍前後に来てしまう）。ページ自身のインク幅をそのままフォントサイズとして
  書き戻すと、書籍全体（旧OCR残存ページ込み）のフォントサイズ基準に対して
  わずかに0.68倍を割り込み、本文ページが丸ごとルビと誤判定されることがある
  （実例: ほんものの魔法使P.11、本文インク幅8.28pt／旧OCR本文12.5pt＝0.662倍で
  しきい値未満になり、本文の縦行が全てルビとして抽出され、assemble_text側で
  本文が1文字ずつの改行に分断された）。`detect_body_size`はテキスト層が
  皆無なPDF（全ページ未処理の初回変換等）では0を返すため、その場合のみ
  最初に処理したページ自身のインク幅中央値をフォールバックの基準値として採用する
- **同一の縦書き列が複数のVisionブロックに分断されると、ブロック間でX座標に
  微小なジッター（実測0.1〜0.5pt）が乗り、jisui2epub.py本体の縦行クラスタリング
  で列が2本に割れることがある**。本体側は文字を「(X中心, Y)」でソートしてから
  隣接文字をY方向のギャップでまとめるため、下側のブロックのX中心が上側より
  わずかに小さいと、ソート順が読み順と逆転し、上側ブロックを処理する時点では
  下側ブロックが「はるか下の別の列」に見えてY方向ギャップしきい値
  （`body_size*1.6`）を超えてしまう（実例: ほんものの魔法使P.12、
  「モプシーはあの…り」と「毛が多いので、事」が別々の縦行に分断）。
  `_snap_column_x`でX中心が近い（列間隔よりは十分小さい・観測ジッターよりは
  十分大きいCOLUMN_SNAP_TOLERANCE=3.0pt以内の）連続文字を同一列とみなし、
  先頭のX座標に完全一致させることで解消した
- **上記のX座標スナップは半角の数字・記号（ノンブル等）を対象・基準から除外する
  必要がある**。ノンブルは本文列のすぐ近くに単独で出現し、かつ半角数字は文字幅が
  本文よりずっと狭い。除外せずにスナップすると、たまたまX中心が近いだけの
  本文列全体がノンブルの狭い幅に上書きされ、本文が丸ごとルビ幅未満と誤判定
  される（実例: ソフロニア嬢、倫敦で恋に陥落するP.15、ノンブル「14」の直後に
  あった本文120字が幅4pt程度に潰されルビとして誤抽出された）。ただし全角化予定の
  半角記号（！？、`_FULLWIDTH_PUNCT`）は本文中の通常の句読点なのでこの除外に
  含めない（含めると単独の孤立記号として列からはみ出す）
- **Visionが返すblock配列の順序は縦書き複数列の読み順と一致しないことがある**
  （列の折り返しで語が2ブロックに分断され、しかもブロック順が前後する）。
  ブロックをX座標降順（縦書きは右の列から読む）に並べ替えてから文字を書き戻す
  ことで解消。見出し（横書き）やノンブル（欄外）はこの並べ替えでは正しい位置に
  来ないが、jisui2epub.py本体の`classify_marginals`が別途座標ベースで再分類する
  ため実害はない
- **【重要・解決済み】ルビ密度が非常に高い本（ほぼ全文に振り仮名がある柏葉幸子
  作品・蘇我氏など）でルビの親文字への紐付けが大きく崩れていた問題は、原因を
  4つ特定してすべて対応した**。正解テキストとルビペア単位（親文字・読み）での
  一致率（`scratchpad/eval_ruby.py`）は、当初の地下室17.8%・霧11.9%・
  蘇我氏0%・黒牢城4.2%から、**地下室90.6%・霧88.5%・蘇我氏73.9%・
  黒牢城76.9%** まで改善（2026-07-16検証）。修正内容:

  **原因1: Visionが1文字だけの極小ルビ（手《て》・引《ひ》）を検出漏れしやすい**
  （実測: 旧OCRには「て」「ひ」が単独行として存在するが、Vision再OCRでは全く
  出現しない）。検出漏れがあると本体のルビ→親文字対応付け（近接優先）が連鎖的に
  ずれる。→ redaction前に旧OCRテキスト層も`analyze_page`で解析しておき
  （旧OCR自己申告の本文サイズ`old_ocr_body_size`を基準に使う）、Visionのルビ
  検出が同じ位置に何もない箇所だけ旧OCR側のルビで補完（`_gapfill_missing_rubies`）。
  補完ルビのフォントサイズは旧OCRの実測セル高さに合わせる（大きすぎると隣の
  ルビと重なり「てひ」のように誤結合される）

  **原因2（最重要・jisui2epub.py本体の修正）: `VLine.cell_height()`がインク高さ
  中央値だったため、書き戻し文字のbbox（フォントメトリクス上、指定fontsizeの
  約1.2倍の高さで描画される）に引きずられて実際の文字送りより大きくなり、
  `attach_rubies`のY座標→文字インデックス変換が累積的にずれていた**
  （例: 地下室で真のピッチ9.94ptに対しcell_height16.4pt→39文字の行の後半では
  10文字以上ずれ、ルビが大きく手前の親文字に紐付く）。→ cell_height()を
  「セルY開始位置の隣接差の中央値（＝真のピッチ）」に変更。1行1スパンの旧OCRでは
  セルが均等分割なので隣接差＝従来値となり回帰しない。1文字1スパンの旧OCR
  （2013年グリック）ではむしろ精度向上を確認（手作業校正版比35.2%→38.0%）。
  この修正により書き戻しフォントサイズが実ピッチより大きくても対応付けが
  狂わなくなり、`max(Vision実測, 旧OCR申告)`の安全側フォントサイズ選択
  （本文がルビ誤判定で消失する回帰の防止）と両立できた。※以前試して撤回した
  「Vision実測値で書き戻す」「fontsizeを1.2で割り引く」等の対症療法は不要になった

  **原因3: ルビの書き戻しフォントサイズが実寸より大きいと、本体のルビ列
  グルーピング（Y方向ギャップがsize×1.7以内なら同一ルビ）の閾値が緩みすぎ、
  親文字1文字を挟んで隣接する別の語のルビ同士が1本に誤結合される**（実例:
  蘇我氏で中大兄皇子《なかのおおえのみこ》+が+中臣鎌足《なかとみのかまたり》の
  2つのルビが結合）。→ ルビはページごとのVision実測幅の中央値（≈4.7pt、
  target_ruby_fontsizeを上限キャップ）で書き戻す。ルビが小さい分には0.68倍
  しきい値判定は安全側に働く

  **原因4: 挿入基準を「ベースライン=Vision字面下端」にしていたため、bbox上端
  （＝セルY開始位置）が実際の字面上端より上にずれ、しかもズレ量がフォント
  サイズに比例する（大きく書き戻す本文と実寸で書き戻すルビでズレが不一致）**。
  この差がY→インデックス変換に+0.3〜0.5文字分の系統バイアスとして乗り、
  親文字範囲が1文字後ろにはみ出す（蘇我氏《そが》←正しくは蘇我《そが》氏）。
  → 挿入ベースラインを`rect.y0 + fontsize`（fontname="japan"のascender=1.0、
  descender=0.2を実測済み）とし、bbox上端をVision字面上端に一致させて解消

  残る不一致はVision自体のルビ読み取り誤り・検出漏れが主（せいさん→いさん、
  ちょうめの「ご」欠落等）で、ジオメトリ起因の崩れはほぼ解消。回帰テスト:
  ほんものの魔法使・伯爵夫人=旧OCR直接変換の出力が修正前後で完全一致、
  グリック=ルビ一致率向上、ソフロニア嬢=Vision経路の本文一致率98.9%を維持、
  全冊で章見出し数の変化なし
- **章見出しのサイズ信号は明示的に保存する必要がある**。一律
  target_body_fontsize で書き戻すと「見出しは本文より大きい活字」という
  情報が消え、本体の is_big 判定（本文比1.18+）が発火しなくなる（実例:
  霧＝旧OCRで見出し16.8pt/本文13.5ptだったのが全ページVision化で章見出し
  全滅）。対策: 縦書き列単位で「インク幅」と「文字送りピッチ」の**両方**が
  ページ中央値の1.15倍以上の列だけ、幅比を保って大きく書き戻す
  （BIG_SIZE_RATIO）。幅単独では判定禁止＝Visionのインク幅は列ぐるみで
  ±20%揺れることがある（霧P.7で本文列が幅比1.23、ピッチ比は1.00で正しく
  棄却）。横書き見出し（岩波型）は列グルーピングに乗らないため未対応
- 挿絵のみのページ（本文なし）はVisionが正しく「テキストなし」を返す
  （旧OCRは記号ノイズを誤検出することがあった。グリックの冒険・霧のむこうの
  ふしぎな町の挿絵ページで確認）
- 全角！／？に対して半角!/? を返すVisionの癖を正規化（`_FULLWIDTH_PUNCT`）
- 既存テキスト層は`add_redact_annot`＋`apply_redactions`（画像・線画は保護）で
  除去してから書き戻す（`remove_ocr_text.py`と同じ手法）
- 一時的なAPIエラー（ServiceUnavailable等）は指数バックオフでリトライ。長時間
  ジョブの中断に備えて`CHECKPOINT_EVERY`(既定20)ページごとに出力PDFを保存し、
  `--start`で再開できる。ただし同じファイルを開いたまま`doc.save()`することは
  PyMuPDFの制約でできない（"save to original must be incremental"）ため、
  一時ファイルに保存してから`os.replace`で置き換える。
  **チェックポイント保存でgarbage回収を使ってはならない**（`_atomic_save`は
  最終保存`final=True`のみgarbage=4）。garbage付き保存はin-memory xrefを
  再番号付けするが、PyMuPDFがinsert_text用にキャッシュするフォントxrefは
  古いままなので、以降のページの`/Resources/Font/japan`が無関係なオブジェクト
  を指し、テキスト抽出がUTF-16BE素通しの文字化けになる（実例: 霧_scansnap、
  チェックポイント直後のp21以降全ページが文字化け。発火は条件依存で、
  再番号付けでフォントxrefがずれた場合のみ壊れる）。壊れたPDFはテキスト
  バイト列自体は正しいので、各ページのフォント参照を正しいフォント
  オブジェクトに付け替えれば再OCRなしで修復できる
- 料金: `document_text_detection`は月1000ユニットまで無料。300〜400ページ程度の
  本なら1冊あたり無料枠に収まる
- **段落の数十字ごとの細かい改行はvision_reocr.py由来ではなく書籍側の特性の
  ことがある**。ほんものの魔法使は「柱・ノンブル除去0行」の未処理（旧OCR）
  ページでも約28〜40字ごとに改行が入る（assemble_textの字下げ判定が
  この本のレイアウトと相性が悪いとみられる）。vision_reocr.py側の不具合
  （上記の縦行分断バグ等）と混同しないよう、疑わしい場合は同じ本の
  未処理ページで同じ症状が出るか比較すること

## docai_reocr.py（前処理: Google Cloud Document AIによる再OCR）

vision_reocr.py のOCRエンジンを Document AI (Enterprise Document OCR,
pretrained-ocr-v2.1) に差し替えた版。無料枠なし・$1.50/1000ページの従量課金
だが精度を優先するユーザー向けの選択肢。

```bash
.venv/bin/pip install google-cloud-documentai
# Document AI API有効化＋課金有効化の上、プロセッサを作成（初回のみ）
.venv/bin/python docai_reocr.py --create-processor
.venv/bin/python docai_reocr.py input.pdf   # 既定出力は <入力>_docai.pdf
# プロセッサは自動検出（環境変数DOCAI_PROCESSORまたは--processorで明示も可）
```

VisionとのA/B実測（同一ページ・同一評価基盤、2026-07-16）:

- ルビペア一致率: 地下室92.0%(Vision90.6) / 霧88.3%(88.5) / 蘇我氏66.7%(73.9) /
  黒牢城76.0%(76.9) — ほぼ互角（蘇我氏の差はn=42での数件差）
- **本文はDocAIが質的にきれい**: 地下室P.4-13の本文精密比較でVision誤り
  14箇所42字（「こし」→「そ」等の内容レベルの脱落・化けを含む）に対し
  DocAI誤り7箇所30字、うち実質誤読は「大→犬」の1字のみ
  （残りはダッシュ／三点リーダの字種揺れとノンブル混入）
- **2エンジンの誤り箇所はほとんど重ならない**（15箇所+8箇所中、共通3箇所のみ）
  → 将来「両方でOCRして不一致箇所だけ人が確認する校正支援」の可能性がある
- DocAIはVisionが検出漏れする1文字の極小ルビ（手《て》等）を自前検出できる
  （旧OCR補完なしでも地下室91.2%を維持）。補完機構は位置照合で重複を防ぐ
  ためそのまま併用している
- 処理速度は約2秒/ページ（Visionの約4倍遅い）。同期APIの画像サイズ上限20MB

実装の要点:

- 書き戻しヒューリスティック（`_snap_column_x`・`insert_invisible_text`・
  `_gapfill_missing_rubies`・`_atomic_save`等）は**vision_reocr.pyからimport
  して共有**。二重実装による将来の乖離を防ぐ。このためvision_reocr.py側は
  `google.cloud.vision` のimportを関数内に遅延化してあり、DocAIだけ使う
  ユーザーはgoogle-cloud-vision未インストールでも動く
- Document AIはシンボルがページ直下のフラット配列で来る（Visionのような
  block→paragraph→word→symbol階層でない）ため、`text_anchor`の文字範囲で
  ブロックに帰属させてから、Visionと同じX座標降順ソート＋列スナップを適用
- シンボルのバウンディングボックスは`bounding_poly.vertices`（ピクセル座標）
  優先、なければ`normalized_vertices`×画像サイズ
- `enable_symbol=True`（OcrConfig）を指定しないとシンボル単位の座標が返らない
- **DocAIはまれに同じルビ字形を二重検出する**（実測: 地下室の住所《じゅうしょ》
  が「じじゅゅううししょ」化）→ 同一文字がX/Y2pt以内で重複したら除去
  （`_dedup_symbols`）。この対処でルビ一致率91.2%→92.0%
- プロセッサは`--create-processor`で作成（OCR_PROCESSORタイプ）、実行時は
  認証JSONのproject_idから自動検出

テスト用サンプル: `temp_sample/` に入力PDF・正解テキスト・公式ePubがある
（グリックの冒険=古いOCR・挿絵多、ほんものの魔法使=新OCR、伯爵夫人=見出し弱、
書を捨てよ=NORMAL/SUPERFINE両画質・エッセイ集、タイム・リープ上下=章頭画像・口絵、
霧のむこうのふしぎな町=高ルビ密度（57個/千字）・入力は`_scansnap.pdf`を使う
